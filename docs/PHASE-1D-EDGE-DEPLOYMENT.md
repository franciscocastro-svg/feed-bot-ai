# Phase 1D — Edge Function deployment safety

## Scope

This phase makes type validation of the two payment/authentication functions a blocking CI gate:

- `auth-email-hook`
- `payments-webhook`

The release manifest at `ops/edge-functions-critical.json` is the source of truth for their entrypoints, Deno configuration, and `verify_jwt` values. It also records the release boundaries: no database migrations, frontend publication, other Edge Function deployments, or VPS runtime changes.

## CI behavior

GitHub Actions installs Deno 2, restores an isolated dependency cache, validates the manifest against `supabase/config.toml`, and runs `deno check --frozen` for both entrypoints. Each function has its own lockfile, so registry changes cannot silently alter a validated build. The check does not deploy or invoke a function and does not need production secrets.

Run locally only when Deno 2 is installed:

```bash
npm run check:edge-functions
```

The normal Node quality gate remains:

```bash
npm run ci
```

The Lovable MCP bundle at `supabase/functions/mcp/index.ts` is generated by the Vite plugin and is excluded from ESLint. Its editable sources under `src/lib/mcp/` remain covered. This prevents generated `var` declarations from weakening the Phase 1C ratchet without taking ownership of Lovable's generated file.

## Deployment boundary

After the commit reaches GitHub and CI passes, deploy only the functions listed in the manifest. This phase has no SQL migration and does not require a frontend publish. Deploying every function would expand the reviewed scope.

## Non-mutating smoke checks

Set the public function base URL without placing any secret in shell history:

```bash
export SUPABASE_FUNCTIONS_URL="https://PROJECT_REF.supabase.co/functions/v1"
```

Check CORS and availability of the email hook without sending an email:

```bash
curl --fail-with-body --silent --show-error \
  -X OPTIONS "$SUPABASE_FUNCTIONS_URL/auth-email-hook" \
  -H 'Origin: https://fluxifeed.com' \
  -H 'Access-Control-Request-Method: POST' \
  -D - -o /dev/null
```

Check rejection at the payment signature boundary without creating a payment or changing the database:

```bash
status="$(curl --silent --output /dev/null --write-out '%{http_code}' \
  -X POST "$SUPABASE_FUNCTIONS_URL/payments-webhook?env=sandbox" \
  -H 'Content-Type: application/json' \
  -H 'stripe-signature: invalid-smoke-signature' \
  --data '{}')"
test "$status" = "400"
```

Expected results are a successful `OPTIONS` response and HTTP `400` for the deliberately invalid Stripe signature. Do not use a real webhook payload, API key, recipient, or customer in these checks.
